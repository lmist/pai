## Problem Statement

Prior authorization for Medicaid is the worst-regulated, worst-tooled corner of an already broken process. Providers — especially FQHCs and safety-net clinics — spend staff hours per patient wrestling with 50-state rules, per-MCO formularies, and opaque denial criteria. Medicaid PA denial rates run ~12.5%, more than double Medicare Advantage's 5.7% (HHS OIG, 2023), and state oversight of denials is weak. As of January 2026, CMS began using AI (WISeR pilot, 6 states) to screen PAs on the payer side, and CMS-0057-F now forces 7-day decision windows. Providers are asymmetrically disadvantaged: payer AI denies faster than humans can respond, and Medicaid clinics are last in line for enterprise EHR-integrated tooling.

The hackathon goal is a working demo for a YC-style investor audience that lands the "this is a company" narrative in 60–90 seconds.

## What Makes This Cool

The hero artifact is **the dial**: a single probability score for whether this specific PA will be approved by this specific MCO for this specific drug, with a decomposable breakdown ("you're at 34% — document a failed metformin trial ≥3 months and you jump to 91%"). It reframes the category: the product isn't "PA form-filling," it's **denial prevention before submission**.

Layered on top of that is per-provider adaptation: the app learns each clinic's PA surface area — top drug classes, MCO mix, denial patterns, documentation style — so the dial gets sharper the longer a clinic uses it. Network effects per-clinic, not per-market.

Closing beat of the demo is the **denial autopsy**: for a PA that was already denied, paste the denial letter → AI cross-references the policy, the chart, and the original submission → produces a color-coded diff showing the exact missing sentence → one-click appeal draft. Emotionally the sharpest moment in the demo.

## Constraints

- Hackathon timebox: weekend-to-long-weekend build.
- No real PHI. Synthetic/de-identified charts only.
- Demo must run reliably offline or on a single laptop.
- No EHR integration. Chart ingest is paste-or-upload, not FHIR.
- Solo or small-team build; stack should match Lou's muscle memory (Next.js + FastAPI + Clerk + Postgres, with Neo4j/Graphiti as an optional flex layer).
- Target runtime for the hero moment (dial render from chart upload): ≤60 seconds end-to-end.

## Premises

1. Hero = the probability dial. All other UX is supporting cast.
2. Provider-side buyer (FQHC / safety-net clinic), not payer-side.
3. Single-state + single-drug-class wedge for the demo (recommend California Medi-Cal + GLP-1s for T2DM, or Texas + pediatric ADHD stimulants).
4. Synthetic/de-identified chart data only. 2–3 prepared scenarios (clear approval, clear denial, borderline-saved-by-the-dial).
5. The dial must be *believable*, not *accurate*. LLM-based rubric scoring against published MCO criteria; rigorous ML calibration is a post-hackathon problem.
6. Missing-evidence drilldown must be specific, not vague ("no documented trial of metformin ≥3 months at max dose"). Specificity is what sells the demo.
7. **Per-provider adaptation.** Tool learns each provider's PA mix, MCO distribution, documentation style, and historical denials. In the demo, this is pre-seeded via a realistic provider profile + simulated submission history. Post-hackathon, adaptation compounds per-clinic — this is the defensibility story.
8. **Regulatory tailwind is the "why now."** WISeR (Jan 2026), CMS-0057-F 7-day windows (Jan 2026), and 2026 state AI-in-PA laws create the pitch opener: "Starting this January, CMS uses AI to deny PAs. We're the AI that helps providers keep up."
9. **Audit trail is non-optional.** Every dial score decomposes into cited MCO criteria + specific chart evidence. Not a polish item — emerging state law requires AI-in-PA tools to be reviewable. Build it in from day one; it's a moat against competitors treating compliance as an afterthought.

## Landscape Findings (Phase 2.75)

- **Layer 1 (conventional wisdom):** PA space is crowded on the commercial/MA side (Cohere Health — acquired by Navina, Rhyme, Myndshft, Availity, Waystar, plus Co:Helm, Anterior, Latent). Received wisdom: you can't build in healthcare without Epic/Cerner integration; automation under ~60% coverage doesn't justify cost to provider CFOs.
- **Layer 2 (what's changed in 2026):** WISeR pilot went live Jan 2026 (NJ, OH, OK, TX, AZ, WA) — payer-side AI is now federally sanctioned. CMS-0057-F forces 7-day Medicaid MCO decision windows. State legislatures are writing disclosure, clinical oversight, and audit requirements for AI-in-PA tooling. Medicaid denial rates (12.5%) are 2x MA (5.7%), with weak state oversight.
- **Layer 3 (eureka):** Incumbents (Cohere, Rhyme) were built for the pre-WISeR world — enterprise distribution, 18-month sales cycles, payer-side contracts. The post-Jan-2026 world is "AI vs. AI," and Medicaid providers are the side with no armor. A provider-side, Medicaid-specific, audit-trail-native tool is the right posture for an environment that didn't exist six months ago. The 60%-automation-threshold trap is dodged by narrow-wedge positioning: promise 90%+ coverage inside one state + one drug class, not 60% across everything.

## Approaches Considered

### Approach A: The Dial Demo (minimal viable)
Single-page Next.js + FastAPI. Upload synthetic chart → LLM structured extraction → score against hand-curated Medi-Cal GLP-1 rubric → render dial + missing-evidence callout. 3 pre-built scenarios.
- Effort: S. Risk: Low. Ships fast, hero moment lands, no moat story.

### Approach B: Adaptive Engine (ideal)
Approach A + provider profile layer ("Valley Community Health — 340B FQHC, Medi-Cal + Molina + HealthNet, top submissions: GLP-1s, SSRIs, ADHD stimulants") + submission history sidebar (5 prior simulated PAs with outcomes) + calibration-over-time chart ("approval rate: 62% → 89% across last 50 PAs"). Per-MCO criteria libraries.
- Effort: M. Risk: Med (adaptation has to look real, not fake).

### Approach C: Denial Autopsy (creative / lateral)
Paste a denial letter → AI reads it → pulls cited policy → cross-references chart → color-coded autopsy (what PA said, what MCO wanted, exact missing sentence) → one-click appeal with win-rate estimate.
- Effort: M. Risk: Med. Not a strong standalone demo (counter-intuitive for 60s), but an outstanding closing beat.

## Recommended Approach

**Approach B as the core demo + Approach C as the closing beat.** ~90-second total runtime.

Narrative arc:
1. **0:00–0:10 — Regulatory hook.** One line: "As of January, CMS uses AI to deny PAs in six states. Providers had no symmetric tool. We built one."
2. **0:10–0:25 — Provider profile reveal.** Valley Community Health's profile loads: their MCOs, drug mix, recent denial patterns. Feels like a real clinic's dashboard.
3. **0:25–0:55 — The dial moment.** Upload a synthetic chart for a GLP-1 PA to Molina Medi-Cal. Dial spins up to 34%. Sidebar shows cited MCO criteria + specific missing evidence. Provider adds one piece (a prior metformin trial note). Dial jumps to 91%. One-click submit.
4. **0:55–1:20 — Calibration graph.** Show approval rate climbing across last 50 simulated PAs as the model adapts to this clinic. This is the moat slide without saying "moat."
5. **1:20–1:50 — Denial autopsy closing beat.** Paste a denial letter. Autopsy appears. Appeal drafts in seconds.
6. **1:50–2:00 — One-liner close.** "Medicaid providers have 12.5% denial rates and zero AI. We change that. Live in one clinic, ready for ten more."

Rationale: Approach A alone leaves the biggest investor question ("why defensible") unanswered. The adaptation layer + the Graphiti/Neo4j-backed provider-knowledge-graph is both the differentiator and a play to Lou's existing strengths. Risk of adaptation feeling fake is managed by pre-seeding realistic synthetic history. The denial autopsy doesn't add much build time on top of B (same RAG pipeline, different input shape) and meaningfully lengthens the "this product has legs" impression.

## Open Questions

- Which wedge — Medi-Cal + GLP-1s, or TX Medicaid + pediatric ADHD stimulants, or Ohio Medicaid + substance use treatment? Each has different demo emotional weight. GLP-1s has name recognition; pediatric ADHD and SUD are more sympathetic and underserved.
- Synthetic chart sourcing: generate via LLM from public clinical vignettes, or adapt from open-source datasets (MIMIC-derived, Synthea)?
- Provider profile: fictional clinic ("Valley Community Health") or explicitly stylized composite of a real FQHC type? Investors respond to specificity; legal exposure is near-zero for synthetic composites.
- Is Graphiti/Neo4j worth the demo-day complexity, or does Postgres + JSONB do the job for the hackathon with the graph story told in slides?

## Success Criteria

- Demo completes the full 90-second arc without failure in 3 consecutive rehearsal runs.
- The dial moment (34% → 91% with specific missing-evidence callout) reliably lands in <10 seconds from the "submit" click.
- Provider profile feels real: a clinician-adjacent person watching would say "yeah, that looks like my clinic."
- At least one audit-trail drill-down is visible on screen (cited criterion + chart citation), even if not explicitly narrated — it sells seriousness.
- Story answers all three investor instinct-questions without Q&A: *what is it, why now, why defensible*.

**Phase 0 (Day 1, first 30 min) — Repo + infra baseline**
0a. `git init && gh repo create lou-medicaid-pa-copilot --private`. Push empty skeleton.
0b. Justfile as single entry point: `demo-up`, `demo-reset`, `demo-seed`, `demo-rehearse`, `test`, `fixtures`.

**Phase 1 (Day 1, morning) — Skeleton + scenarios**
1. Next.js + FastAPI + Postgres skeleton via existing templates. Postgres + JSONB for provider profile, submissions, criteria. (Neo4j/Graphiti deferred — graph story lives in slides.)
2. Write 3 synthetic patient charts (clear approval, clear denial, borderline) as Markdown/PDF. Write 1 synthetic denial letter for the autopsy scene.
3. Build Medi-Cal GLP-1 criteria rubric as a **Python module** (`backend/rubrics/medi_cal/glp1.py`), not JSON. One `Criterion` dataclass per rule with a typed `check(extraction) -> CriterionResult` method. Hand-curate from published Medi-Cal formulary docs. Type-checked, unit-testable, one-file-per-(state,drug-class) for post-hackathon expansion.

**Phase 2 (Day 1, afternoon) — The dial**
4. Chart ingestion endpoint: PDF/Markdown → structured extraction via GPT-4-class model with JSON schema output (age, dx, meds, prior trials, labs).
5. Scoring endpoint: structured chart + rubric → probability score + per-criterion breakdown + missing-evidence list. **FIXTURE_MODE=strict**: hash `(chart_id, mco_id, drug_class, rubric_version)` → cached JSON in `backend/fixtures/scores/`. Demo runs read-only from cache; misses raise loudly. Response shape is flat with addressable criteria (id, status, required, evidence, chart_citation{page,excerpt}) so the dial UI renders directly with no intermediate layer. Use OpenAI structured outputs (`response_format: {type: "json_schema"}`) with a strict Pydantic `ChartExtraction` schema for the live path.
6. Dial UI component: animated SVG, citation drill-down on click, missing-evidence sidebar with "apply fix" button that mutates the chart and re-scores.

**Phase 3 (Day 2, morning) — Provider adaptation layer**
7. Provider profile model (Postgres): clinic metadata, MCO mix, top drug classes, historical submissions, per-drug-class approval rates.
8. Submission history sidebar: paginated list of last 50 simulated PAs with outcomes. Seed deterministically.
9. Calibration graph: line chart of approval rate over submission count. Pre-seed the trend.

**Phase 4 (Day 2, afternoon) — Denial autopsy**
10. Denial letter ingestion + policy-citation extraction (RAG over Medi-Cal policy docs).
11. Autopsy view: three-pane diff (submission | policy requirement | chart evidence) with color coding.
12. One-click appeal draft: LLM generates appeal with citations.

**Phase 4.5 — Tests (required, parallel with each phase as components land)**

- `backend/tests/test_rubrics_glp1.py` — every Criterion: met, unmet, ambiguous/missing-field, numeric boundary (A1c exactly 7.5%).
- `backend/tests/test_score_fixture_mode.py` — strict-mode cache hit returns cached; strict-mode cache miss RAISES (not falls through).
- `backend/tests/test_extract_schema.py` — valid PDF extracts to Pydantic model; malformed PDF raises with clear error.
- `backend/tests/test_seed_integrity.py` — after `demo-reset && demo-seed`, `/api/profile/valley-community` returns exactly 50 submissions with a non-empty approval-rate trend. (Guards the critical silent-failure mode where calibration graph renders empty.)
- `tests/evals/glp1_scoring.yaml` — 10 hand-graded scenarios (3 strong approval, 3 strong denial, 4 borderline). Each asserts a probability band. Runs in CI and before every dress rehearsal.
- `tests/e2e/demo-rehearse.spec.ts` — Playwright walks the full 90-second arc and asserts each beat: profile reveal, dial 34%, apply-fix, dial 91%, submit, calibration graph non-empty, autopsy three-pane diff, appeal draft populates. **IRON RULE: passes before every rehearsal.**

**Phase 5 (Day 3) — Rehearsal + polish**
13. End-to-end rehearsal: run `just demo-rehearse` (Playwright) until three consecutive green runs. This is also the offline-fallback verifier — run with `OPENAI_API_KEY=unset` to confirm fixture mode is airtight.
14. Demo-day environment: fixtures baked into repo, stack spins up with `just demo-up` with no external API calls during the rehearsed path.
15. Audit-trail visibility pass: make sure cited criteria + chart citations are on-screen during the dial moment — clickable per-criterion drill-down is on by default, not tucked into a side modal.
16. **Autopsy cut-line decision**: if Phase 4 autopsy is not green by Sunday noon, cut from live demo. Keep it in slides as "here's what happens next." Dial moment rehearsal takes priority — it's the only beat that must land.
